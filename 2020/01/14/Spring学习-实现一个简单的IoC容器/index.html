Spring学习-实现一个简单的IoC容器

发表于 2020-01-14

缘起

平时工作上最常使用的就是Spring框架, Spring 的 IoC 实在太方便了！ 只需要告诉 Spring 我需要什么，IoC容器就会自动注入，我们直接调用就完事了。不仅是使用方便，而且实现了对象间的解耦。Spring 之前的对象，多重依赖关系特别严重，这就导致了耦合关系不仅会出现在对象与对象之间，也会出现在软件系统的各模块之间，以及软件系统和硬件系统之间。就像齿轮，一个紧挨着一个。而 Spring 的 IoC 容器，就是中间位置的 “第三方”，使得对象间没有了耦合关系，齿轮之间的传动全部依靠“第三方”了，全部对象的控制权全部上缴给“第三方”IoC容器，所以，IoC容器成了整个系统的关键核心，它起到了一种类似“粘合剂”的作用，把系统中的所有对象粘合在一起发挥作用，如果没有这个“粘合剂”，对象与对象之间会彼此失去联系，这就是有人把IoC容器比喻成“粘合剂”的由来。

使用起来简单，但是它深层的原理我之前并不了解。这种滋味可不好，所以在工作之余，参考了一些资料和代码，探究一下 Spring 的实现原理， 能够揭下这位“美女”脸上的面纱，真是激动呢！

实现一个简单的IoC

首先，我们定义一个接口 BeanFactory 标识一个 IoC 容器。通过 getBean(String) 方法来 获取一个对象。

好的，perfect，下一步是啥？ 我们的容器还空空如也，需要给容器注入Bean。问题又来了，有哪些Bean？Bean的名字是什么？ 它的类型是什么？ 它的属性赋予了哪些值或者引用？ So 我们要再创建一个 BeanDefinition 类。

好的，Bean的“模型”已经做好了，现在我们定义一个BeanFactory的抽象类。这个抽象类中定义了一个BeanDefinition的Map用来存放Bean模型，而且定义了储存的方法registerBeanDefinition。实现了getBean方法，当然没有具体逻辑：）

此时 我们有了加工Bean的工厂，有了模具，但是模具是怎么产生的呢？是的 我们需要从”需求清单“里得到模具。

于是我们再定义Resource接口，它是spring内部定位资源的接口。以及实现类，获得url的输入流。还有资源的载入器 ResourceLoader。

接下来，我们需要根据”需求“定制”模具“，定义一个BeanDefinitionReader 接口，从配置中读取BeanDefinition。

BeanDefinitionReader 的抽象类，其中定义了一个存储BeanDefinition的Map仓库，还引进了之前定义的资源加载器ResourceLoader。接下来我们再实现一个读取XML配置文件的实现类XmlBeanDefinitionReader。

使用xml配置之后，似乎里我们熟知的Spring更近了一步！但是现在有一个大问题没有解决：我们无法处理bean之间的依赖，无法将bean注入到bean中，所以它无法称之为完整的IoC容器！如何实现呢？我们定义一个BeanReference，来表示这个属性是对另一个bean的引用。这个在读取xml的时候初始化，并在初始化bean的时候，进行解析和真实bean的注入。

最后，我们再实现BeanFactory的实现类AutowireCapableBeanFactory。可自动装配内容的BeanFactory。它终于实现了getBean的具体逻辑。循环遍历BeanDefinitionMap，根据BeanDefinition的BeanClass来创建Bean，根据ropertyValue 对象来给Bean注入属性。

好的，此时此刻，我们终于完成了一个简单的IoC容器，哈哈哈，Spring的美丽容颜的一角终于被我们窥视到了！
